# 邻里互助项目数据库表结构初稿

## 1. 文档目的

本文档用于输出邻里互助项目的数据库表结构初稿，目标是：

1. 明确哪些表直接复用芋道框架基础表
2. 明确哪些表是邻里互助业务新增表
3. 明确核心主键、状态字段、关联关系、索引建议
4. 让后端、管理端、接口设计不再靠猜字段

本文档是“数据库结构初稿”，不是最终建表 SQL。正式落库时应基于本稿继续输出：

- 业务建表 SQL
- 数据迁移脚本
- 索引优化脚本

当前已同步产出第一版业务建表文件：

- [01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql)
- [03_lb_seed_data.sql](../ruoyi-vue-pro/sql/mysql/03_lb_seed_data.sql)

## 2. 设计原则

### 2.1 总原则

数据库必须遵守：

- 框架基础表复用
- 业务表独立扩展
- 状态机前置
- 日志留痕
- 金额可对账
- 规则可配置

### 2.2 命名原则

- 芋道框架基础表：保留框架原命名
- 邻里互助业务表：统一使用 `lb_` 前缀
- 关联表：`*_rel`
- 记录表：`*_record`
- 流水表：`*_flow`
- 规则表：`*_rule`

### 2.3 字段原则

所有业务主表建议至少包含：

- `id`
- `creator`
- `create_time`
- `updater`
- `update_time`
- `deleted`

涉及状态机的表必须包含：

- `status`

涉及审核流的表建议包含：

- `audit_status`
- `audit_remark`
- `audit_by`
- `audit_time`
- `reject_reason`

## 3. 芋道框架基础表处理原则

## 3.1 结论

数据库设计时必须考虑芋道框架已有基础表，不能把这些能力再重复造一套。

最终实施要求：

- 框架基础表：以 `ruoyi-vue-pro` 官方初始化 SQL 为准
- 邻里互助业务表：只补充项目业务域
- 支付：优先复用框架支付模块表
- 权限：优先复用框架权限体系表

## 3.2 需要直接复用的基础表分组

以下表分组不应在邻里互助业务中重复建设。

### 3.2.1 系统权限与后台账号

用途：

- 管理端用户
- 角色
- 菜单
- 数据权限
- 部门 / 岗位

典型表：

- `system_users`
- `system_role`
- `system_menu`
- `system_role_menu`
- `system_user_role`
- `system_dept`
- `system_post`

说明：

- 这套用于后台管理账号
- 不用于承载 App 普通用户主体

### 3.2.2 字典与系统配置

用途：

- 字典值
- 系统参数
- 配置中心

典型表：

- `system_dict_type`
- `system_dict_data`
- `system_config`

说明：

- 订单状态、计价方式等前端下拉可部分走字典
- 但核心状态机仍应在代码枚举与业务规则中统一定义

### 3.2.3 基础设施

用途：

- 文件上传
- 定时任务
- 日志

典型表：

- `infra_file`
- `infra_job`
- `infra_job_log`
- `system_operate_log`
- `system_login_log`

说明：

- 身份证、资质、交付凭证等文件，优先走框架文件存储能力
- 审核、解锁、提现、退款等行为日志，优先复用操作日志体系并补业务日志

### 3.2.4 支付模块

用途：

- 支付订单
- 退款订单
- 渠道管理
- 应用配置

典型表：

- `pay_app`
- `pay_channel`
- `pay_order`
- `pay_refund`
- `pay_notify_log`

说明：

- 这些表优先复用框架 `yudao-module-pay`
- 邻里互助业务层不要再重复建一套通用支付底座

## 3.3 基础表与业务表边界

必须明确以下边界：

- 管理端账号用框架 `system_users`
- App 用户单独建邻里互助业务表
- 支付订单复用 `pay_order`
- 退款订单复用 `pay_refund`
- 钱包、分账、提现、银行卡由邻里互助业务表承接

## 4. 邻里互助业务表总览

## 4.1 模块划分

建议邻里互助业务表按以下模块落地：

1. 用户与认证
2. 服务商与入驻
3. 服务类目与服务点
4. 订单与拆分引擎
5. 钱包、分账、提现
6. 风控
7. 投诉、申诉、评价、信用
8. 推广员与佣金
9. 区域合作商
10. 消息模板与消息记录

## 4.2 业务表数量建议

当前版本建议先控制在核心表落地，不要一开始把所有统计宽表都建完。

建议第一版重点落下面这些主表和关键关联表。
