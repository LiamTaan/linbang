# 邻里互助初始字典与类目种子数据

## 1. 文档目的

本文档用于说明邻里互助项目第一版初始化数据的范围、来源和落库方式。

当前已同步提供可执行 SQL：

- [03_lb_seed_data.sql](../ruoyi-vue-pro/sql/mysql/03_lb_seed_data.sql)

## 2. 当前种子数据覆盖范围

本次先初始化项目最关键、最容易影响联调的一批业务数据：

1. 服务类目
2. 敏感词
3. 风控规则
4. 信用分规则
5. 分账规则
6. 消息模板

这些数据全部落在邻里互助业务表，不依赖对芋道基础表结构做额外猜测。

## 3. 服务类目初始化

当前初始类目按现有原型图和功能表归纳，先给一套够跑业务的默认树：

- 家电维修
  - 水电维修
  - 空调维修
  - 冰箱维修
  - 洗衣机维修
- 搬家运输
  - 同城搬家
  - 货物搬运
  - 家具运输
- 家政保洁
  - 日常保洁
  - 深度保洁
  - 开荒保洁
- 施工装修
  - 墙面刷漆
  - 瓷砖铺设
  - 木工安装
- 跑腿代办
  - 同城取送
  - 代买代办
- 家具安装
  - 床柜安装
  - 灯具安装

说明：

- 这套不是最终商业全量品类
- 但足够支撑当前原型图和一版需求开发
- 后续类目新增由管理端“服务类目管理”页面维护

## 4. 敏感词初始化

当前初始敏感词分三类：

- 联系方式词
  - 微信
  - vx
  - v信
  - 手机号
  - 电话
  - QQ
- 导流词
  - 二维码
  - 私聊
  - 加我
  - 外链
- 风险词
  - 线下交易
  - 转私账

说明：

- 这是第一版基础词库
- 后续由管理端“敏感词管理”维护
- OCR 命中的词也按这套词库做统一拦截

## 5. 风控规则初始化

当前默认风控规则直接对应总稿里的全局规则：

- 自动拆分金额阈值：`200`
- 自动拆分数量阈值：`5`
- 单元最大金额：`200`
- 抢单倒计时秒数：`60`
- 流单超时时长分钟：`5`
- 连续取消限制次数：`3`
- 取消限制时长分钟：`1440`
- 大额单保证金阈值：`1000`
- 自动评价小时数：`24`
- 申诉有效期天数：`7`

另外补了一条强制拆分类目规则：

- 施工装修
- 搬家运输

## 6. 信用分规则初始化

当前初始信用规则：

- 收到好评：`+1`
- 自动默认好评：`+1`
- 有效投诉：`-5`
- 恶意取消：`-3`
- 逃单：`-20`
- 实名通过：`+2`

这套规则可以先支撑信用分展示、信用等级逻辑和基础处罚场景。

## 7. 分账规则初始化

当前先给一套能跑流程的默认分账规则：

- 一线城市默认规则：平台 `8%`
- 下沉城市默认规则：平台 `5%`
- 施工装修类默认规则：平台 `8%`
- 搬家运输类默认规则：平台 `7%`

当前 SQL 中默认按：

- 服务商分成
- 平台分成
- 合作商分成
- 推广员分成
- 个税代扣比例

进行存储。

说明：

- 这套规则先保障流程跑通
- 后续实际商务结算比例可由管理端“分账规则”页面调整

## 8. 消息模板初始化

当前初始化了以下站内消息模板：

- 支付成功通知
- 接单成功通知
- 提现申请通知
- 提现到账通知
- 退款成功通知
- 证件到期提醒
- 投诉提交通知

说明：

- 第一版先以站内消息模板为主
- 微信模板消息、短信模板后续在对接真实渠道时再补渠道配置

## 9. 为什么暂时不直接初始化框架字典表

当前没有直接去写 `system_dict_type`、`system_dict_data` 的 SQL，原因是：

- 这些表要严格以 `ruoyi-vue-pro` 实际库结构为准
- 先把邻里互助业务表种子数据落稳更安全
- 避免因为基础表字段差异导致初始化脚本导入失败

后续如果你需要，我可以在拉下芋道实际仓库结构后，继续补一份：

- `system_dict_*` 适配版初始化脚本

## 10. 当前开发价值

这份种子数据的价值在于：

1. 管理端下拉和类目树不再空白
2. 风控规则有默认值，拆单和抢单逻辑可直接读取配置
3. 敏感词拦截不再从零开始
4. 消息模板、信用规则、分账规则都有默认配置

也就是说，现在已经不只是“表有了”，而是“表里有第一批能跑业务的数据了”。

## 11. 下一步建议

当前最适合继续补的是：

1. 接口清单与字段契约
2. 芋道基础字典表适配版初始化脚本

如果后端先开工，我更建议优先继续做第 1 项。
