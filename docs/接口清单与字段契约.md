# 邻里互助接口清单与字段契约

## 1. 文档目的

本文档用于说明当前项目接口契约的落地方式，并指出后续开发时应以哪份结构化文件为准。

当前已同步产出：

- [api-contract-core.yaml](../spec/api-contract-core.yaml)

## 2. 当前契约覆盖范围

本次接口契约优先覆盖第一批最容易联调出问题的核心接口：

1. 用户资料与实名认证
2. 服务商入驻
3. 服务类目
4. 订单创建、分页、详情、接单、交付凭证
5. 钱包、提现、银行卡
6. 投诉、申诉、评价
7. 管理端用户分页、审核、订单分页、异常处理
8. 提现审核、退款审核、风控规则、敏感词、投诉申诉处理

## 3. 为什么要用结构化契约

之前容易出现的问题是：

- 前端自己猜字段
- 后端自己起字段名
- 管理端表格再按第三套口径写

所以这次不只给接口路径，还把以下内容一起定住：

- 请求字段
- 返回字段
- 分页结构
- 审核字段
- 列表与详情字段口径

## 4. 使用规则

后续开发必须遵守：

1. 接口骨架以 [openapi-structure.md](../spec/openapi-structure.md) 为范围入口
2. 字段契约以 [api-contract-core.yaml](../spec/api-contract-core.yaml) 为当前实现标准
3. 状态值以 [business-rules.yaml](../spec/business-rules.yaml) 为准
4. 表字段来源以 [01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql) 为准

## 5. 当前已经固定的核心约束

### 5.1 路径前缀

- 用户端：`/app-api`
- 管理端：`/admin-api`

### 5.2 响应结构

- 普通接口：`CommonResult<T>`
- 分页接口：`CommonResult<PageResult<T>>`

### 5.3 审核接口字段

审核类接口统一使用：

- `auditStatus`
- `auditRemark`
- `rejectReason`

### 5.4 订单与拆单

当前已定住：

- 订单详情返回必须包含价格明细、附件、拆分单元
- 管理端订单分页必须包含支付状态、订单状态、拆分状态、风险标记、投诉标记
- 单元解锁必须走单独接口

## 6. 当前开发价值

这份契约的价值是：

1. 前端不再自己猜字段
2. 管理端表格和详情页字段可以直接对照实现
3. 后端接口不会再出现“先临时返回，后面再改”
4. 会话切换后，Codex 也能继续遵循同一套字段口径

## 7. 下一步建议

当前最适合继续做的是：

1. 对照 `api-contract-core.yaml` 开始建后端 BO / VO / Controller
2. 对照 [后端模块开发任务清单.md](./后端模块开发任务清单.md) 分模块实施
3. 对照 `api-contract-core.yaml` 开始建管理端页面字段绑定
4. 如需更严谨，再继续补一份完整 `openapi.yaml`

如果你继续让我往前推，我下一步最合适的是直接开始：

- `按模块正式进入后端与管理端开发`
