## 4. 项目本质判断

这个项目不是普通的信息展示平台，而是一个带有交易闭环的平台系统，核心特征包括：

- 多角色：普通用户、服务商、推广员、区域合作商、平台管理员
- 强交易：发单、支付、抢单、流单、拆单、完工、结算、退款
- 强合规：实名认证、资质审核、协议确认、证件到期控制
- 强资金：托管、冻结、可提现、分账、佣金、退款
- 强风控：防戏单、防刷单、防逃单、防导流、防违规推广
- 强后台：审核、规则配置、投诉处理、数据看板、日志审计

结论很明确：后端和管理端是项目底座，用户端只是业务触点。

## 5. 框架结论

### 5.1 统一用哪套

项目统一走芋道体系，采用以下官方组合：

- 后端：`ruoyi-vue-pro`
- 管理端：`yudao-ui-admin-vue3`

这里名字容易引起误解，但要明确：

- `ruoyi-vue-pro` 是芋道官方主线单体后端项目
- `yudao-ui-admin-vue3` 是芋道官方 Vue3 管理端项目

所以这不是“两套框架混用”，而是芋道官方生态内的一套标准组合。官方资料可参考：

- [ruoyi-vue-pro 开发指南](https://doc.iocoder.cn/)
- [ruoyi-vue-pro GitHub](https://github.com/YunaiV/ruoyi-vue-pro)
- [yudao-ui-admin-vue3 GitHub](https://github.com/yudaocode/yudao-ui-admin-vue3)

### 5.2 当前不要选什么

当前阶段不建议直接使用：

- `yudao-boot-mini`
- `yudao-cloud-mini`

原因：

- `yudao-boot-mini` 更偏轻量起步版，当前项目后续扩展面较大
- `yudao-cloud-mini` 偏微服务，当前阶段会显著增加复杂度

### 5.3 当前架构建议

建议当前采用：

- 一个 `ruoyi-vue-pro` 后端工程
- 一个 `yudao-ui-admin-vue3` 管理端工程
- 一个独立的用户端前端工程

## 9. 地图与支付方案

### 9.1 地图方案

地图能力统一采用：

- `高德地图`

落地方式：

- 用户端前端：高德 `JS API`
- 后端服务：高德 `Web 服务 API`

优先接入能力：

- 定位
- 地址选择
- 地理编码
- 逆地理编码
- 距离计算
- 周边检索
- 行政区划解析

数据库建议统一保存：

- `province`
- `city`
- `district`
- `street`
- `detail_address`
- `longitude`
- `latitude`
- `adcode`

### 9.2 支付方案

支付方式统一支持：

- `微信支付`
- `支付宝支付`
- `银行卡`

业务建议：

- 下单支付：优先微信、支付宝
- 提现与绑卡：使用银行卡

后端需统一抽象：

- 支付订单
- 支付回调
- 退款
- 分账
- 提现
- 银行卡管理

Codex 实现要求：

- 直接复用芋道框架自带的 `yudao-module-pay`
- 在框架支付模块基础上配置 `微信支付`、`支付宝支付` 参数
- 不要重复自研通用支付底座，不要额外再造一套支付中心轮子

本项目支付落地建议：

- 订单支付、支付单查询、支付回调、退款回调，优先走 `yudao-module-pay`
- 钱包、分账、提现、银行卡管理作为业务模块扩展，与支付模块做业务关联
- 如果后续要支持银行卡直接支付，再单独评估对应支付通道；当前需求以微信、支付宝支付 + 银行卡提现为主

开发前仍需补齐：

- 微信支付商户号
- 支付宝商户号
- API 证书 / 密钥
- 回调地址
- 退款配置

接口设计统一按芋道框架的 Restful 规范走，Controller 层风格参考框架自带的 `System` 模块实现。

Codex 开发要求：

## 12. 拆分引擎规则

拆分引擎是本项目核心业务，不允许只做一个“拆分标记字段”草率带过，必须明确以下规则。

### 12.1 拆分触发条件

默认支持以下触发条件，具体阈值后台可配：

- 订单金额达到阈值
- 数量达到阈值
- 指定服务类目强制拆分
- 工程/多人协作类订单强制拆分
- 用户主动勾选拆分

### 12.2 单元金额上限

- 默认单个拆分单元金额上限：`<= 200 元`
- 超出上限时，必须继续细分为多个单元
- 单元金额上限应进入风控规则配置项

### 12.3 拆分方式

支持以下拆分模式：

- 按工序拆分
- 按进度拆分
- 按数量拆分
- 按内容拆分
- 按多人协作拆分

### 12.4 锁单逻辑

- 上一单元未完成时，下一单元默认不可接单
- 上一单元未验收/未确认/存在申诉时，后续单元继续锁定
- 管理端可人工解锁，但必须留日志
- 人工解锁只开放给有权限角色

### 12.5 单元状态机

建议单元状态至少包含：

1. 待生成
2. 待接单
3. 已接单
4. 服务中
5. 待验收
6. 已完成
7. 申诉中
8. 已退款
9. 已关闭

主订单状态必须和单元状态联动：

- 所有单元完成，主订单才可完成
- 任一单元申诉中，主订单进入售后中
- 任一单元退款完成，主订单金额和状态需重新计算

### 12.6 拆分引擎开发要求

- 拆分规则放后台配置，不写死
- 拆分结果要可回溯
- 每次拆分需记录规则命中依据
- 拆分、锁单、解锁、关闭都必须留日志

- 业务表结构单独维护迁移脚本

## 19. 测试策略建议

### 19.1 功能测试优先级

优先测试：

- 注册登录
- 实名认证
- 入驻审核
- 下单支付
- 抢单
- 拆分订单
- 提现
- 退款
- 投诉与申诉

### 19.2 资金模块测试

重点验证：

- 支付成功回调幂等
- 退款回调幂等
- 分账记录与订单金额一致
- 提现申请与钱包余额一致

- 申诉中单元是否阻止后续单元继续推进

### 19.4 管理端测试

- 审核流程权限

## 21. 当前开发结论

当前项目已经满足“可以开始后端和管理端开发”的条件。

最合适的开发顺序：

1. 搭建 `ruoyi-vue-pro` 后端工程
2. 搭建 `yudao-ui-admin-vue3` 管理端工程
3. 初始化芋道框架基础表与支付模块基础表
4. 执行 [01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql) 与 [03_lb_seed_data.sql](../ruoyi-vue-pro/sql/mysql/03_lb_seed_data.sql)
5. 按 [api-contract-core.yaml](../spec/api-contract-core.yaml) 先实现核心后端接口
6. 管理端与用户端页面再接入真实接口，不允许先做长期 mock 壳子
7. 最后补统计、推广、合作商、消息等同版次能力

当前已经具备的结构化开发依据：

- 业务规则：[spec/business-rules.yaml](../spec/business-rules.yaml)
- 接口骨架：[spec/openapi-structure.md](../spec/openapi-structure.md)
- 核心字段契约：[spec/api-contract-core.yaml](../spec/api-contract-core.yaml)
- 数据表设计：[docs/数据库表结构初稿.md](./数据库表结构初稿.md)
- 建表 SQL：[01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql)
- 种子数据：[03_lb_seed_data.sql](../ruoyi-vue-pro/sql/mysql/03_lb_seed_data.sql)
- 模块任务清单：[docs/后端模块开发任务清单.md](./后端模块开发任务清单.md)

Codex 开发执行要求再明确一次：

- 后端仓库：`https://github.com/YunaiV/ruoyi-vue-pro`
