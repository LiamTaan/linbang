# 邻里互助后端模块开发任务清单

## 1. 文档目标

本清单用于把“规则、表结构、接口契约”继续压实成可直接开发的模块任务。

开发时必须同时遵守：

- [AGENTS.md](../AGENTS.md)
- [spec/business-rules.yaml](../spec/business-rules.yaml)
- [spec/openapi-structure.md](../spec/openapi-structure.md)
- [spec/api-contract-core.yaml](../spec/api-contract-core.yaml)
- [docs/数据库表结构初稿.md](./数据库表结构初稿.md)
- [01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql)

## 2. 开发前置

先完成以下基础动作，再进入业务模块：

1. 拉取后端仓库 `https://github.com/YunaiV/ruoyi-vue-pro`
2. 初始化框架基础表，不重复创建 `system_*`、`infra_*`、`pay_*`
3. 执行 [01_lb_business_schema.sql](../ruoyi-vue-pro/sql/mysql/01_lb_business_schema.sql)
4. 执行 [03_lb_seed_data.sql](../ruoyi-vue-pro/sql/mysql/03_lb_seed_data.sql)
5. 确认支付复用 `yudao-module-pay`
6. 确认地图统一使用高德

## 3. 模块实施顺序

当前唯一版本内，推荐按下面顺序开发：

1. `member`
2. `merchant`
3. `order`
4. `wallet`
5. `review`
6. `risk`
7. `promote`
8. `partner`
9. `message`
10. `dashboard`

说明：

- 这是同一版需求下的实现顺序，不是产品拆版本
- 前 1 到 6 是主交易链路与联调高风险模块
- 后 7 到 10 仍属于当前版，只是排在主链路之后

## 4. P0 模块任务

### 4.1 `member` 模块

涉及业务表：

- `lb_user`
- `lb_user_real_name`
- `lb_user_qualification`
- `lb_user_role_apply`
- `lb_user_address`

首批用户端接口：

- `GET /app-api/member/user/profile`
- `POST /app-api/member/real-name/create`
- `GET /app-api/member/real-name/get`
- `GET /app-api/member/address/page`
- `POST /app-api/member/address/create`
- `PUT /app-api/member/address/update`
- `DELETE /app-api/member/address/delete`

首批管理端接口：

- `GET /admin-api/member/user/page`
- `GET /admin-api/member/real-name/page`
- `POST /admin-api/member/real-name/audit`
- `GET /admin-api/member/qualification/page`
- `POST /admin-api/member/qualification/audit`

管理端页面：

- 用户列表
- 实名认证审核
- 资质认证审核
- 身份申请审核

完成标准：

- 用户资料、实名、地址全部真实落库
- 审核接口统一使用 `auditStatus`、`auditRemark`、`rejectReason`
- 管理端分页、详情、审核动作全部接真实接口

## 5. 同版次后续模块

以下模块仍属于当前版，但排在主链路之后：

- `promote`：推广员、邀请码、佣金明细
- `partner`：区域合作商、辖区、价格申报、初审权限
- `message`：站内信、短信、模板、发送记录
- `dashboard`：平台工作台、财务统计、订单统计、用户统计

这些模块的页面不能只做壳子，也必须遵守真实接口接入标准。

## 6. 统一交付标准

每个模块开发完成，至少满足：

1. Entity / BO / VO / Mapper / Service / Controller 成套存在
2. 管理端分页、详情、审核、状态流转全部走真实接口
3. 页面字段与 [api-contract-core.yaml](../spec/api-contract-core.yaml) 一致
4. 状态值与 [business-rules.yaml](../spec/business-rules.yaml) 一致
5. 不出现长期 mock 页面、死按钮、临时字段名

## 7. 下一步建议

下一步最合适的动作不是继续写散文档，而是：

1. 拉下芋道官方后端与管理端仓库
2. 初始化框架基础库
3. 按本清单开始实现 `member`、`merchant`、`order`、`wallet`
4. 管理端页面同步对照真实接口开发
