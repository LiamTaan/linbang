# 邻里互助项目 AGENTS

本文件是本仓库的项目级强约束。所有进入本仓库的 Codex 会话、开发者和自动化任务，必须优先阅读并遵守本文件。

## 1. 项目定位

- 项目中文名：`邻里互助`
- 项目英文名：`LinBang`
- 项目类型：本地生活 / 邻里互助 / 交易撮合平台

这是一个强交易、强审核、强资金、强风控的平台系统，不是简单信息展示系统。

## 2. 仓库与框架

Codex 注意：

- 后端必须基于 `https://github.com/YunaiV/ruoyi-vue-pro`
- 管理端必须基于 `https://github.com/yudaocode/yudao-ui-admin-vue3`

绝对禁止：

- 禁止拉错成 `yudao-boot-mini`
- 禁止拉错成 `yudao-cloud-mini`
- 禁止拉第三方 `ruoyi-vue-pro` 分叉仓库
- 禁止自行起另一套后台框架

说明：

- `ruoyi-vue-pro` 是芋道官方主线单体后端项目
- `yudao-ui-admin-vue3` 是芋道官方 Vue3 管理端项目
- 这两者是本项目唯一允许的标准组合

## 4. 开发顺序强约束

禁止先堆壳页面再补规则。

必须按以下顺序推进：

1. 业务规则
2. 数据模型
3. 接口契约
4. 后端实现
5. 管理端 / 用户端页面接入真实接口
6. 联调
7. 测试

## 5. 页面交付标准

页面不能只做成“架子”。

一个页面只有同时满足以下条件，才可视为完成：

1. 接入真实接口
2. 使用真实字段渲染
3. 覆盖真实业务状态
4. 覆盖真实权限控制
5. 支持真实错误提示
6. 不依赖长期 mock 数据

以下情况不算完成：

- 页面能打开但是假数据
- 表格能展示但按钮无真实动作
- 弹窗能打开但不改变业务状态
- 详情页字段与后端真实返回不一致

## 6. 接口与分层规则

后端必须严格按芋道标准分层：

- Entity
- BO
- VO
- Mapper
- Service
- Controller

接口路径必须分离：

- 用户端：`/app-api/*`
- 管理端：`/admin-api/*`

接口风格必须参考芋道 `System` 模块：

- 分页接口
- 详情接口
- 新增接口
- 修改接口
- 删除接口
- 导出接口
- 审核接口

禁止：

- 自造另一套响应结构
- Controller 直接拼业务流程
- 用 Map 充当主业务对象

## 7. 支付与地图强约束

支付能力：

- 微信支付
- 支付宝支付
- 银行卡提现/绑卡

支付实现要求：

- 必须复用芋道框架自带 `yudao-module-pay`
- 禁止重复自研通用支付中心
- 钱包、分账、提现作为业务模块扩展

地图能力：

- 统一使用高德地图

落地要求：

- 前端：高德 `JS API`
- 后端：高德 `Web 服务 API`

## 8. 核心业务规则不可绕过

必须遵守 `spec/business-rules.yaml` 中的以下规则：

- 用户与服务商角色边界
- 订单状态机
- 拆分单元状态机
- 支付、退款、提现状态机
- 区域合作商数据权限
- 风控配置项
- 分账规则配置项

特别说明：

- 拆分引擎是本项目核心功能，不能简化成单字段标记
- 主订单与单元订单必须联动
- 锁单、解锁、退款、申诉必须留日志

## 9. 联调规则

联调前必须先确认：

1. 接口路径已定
2. 请求字段已定
3. 返回字段已定
4. 状态值已定
5. 字典值已定

禁止：

- 前端按猜测接接口
- 管理端和用户端各用一套状态文案
- 后端先返回临时字段名，后面再改

## 10. 会话工作规则

新会话进入本仓库后，默认第一步必须：

1. 读取本 `AGENTS.md`
2. 读取 `spec/business-rules.yaml`
3. 读取 `spec/openapi-structure.md`
4. 读取 `spec/api-contract-core.yaml`

如需新增模块或页面，必须先检查：

- 是否已有对应业务对象
- 是否已有对应状态机
- 是否已有对应接口前缀与命名规范

如果没有，先补规则，再写代码。

## 11. 当前阶段最优先产物

当前最优先推进顺序：

1. 全局业务规则一致性校验
2. 数据库表结构与业务 SQL
3. 接口清单与字段契约
4. 后端核心模块实现
5. P0 页面接入真实接口

如果规则、表结构、接口清单未完成，不建议大面积扩展页面开发。
如果后端真实接口未完成，也不建议继续堆长期 mock 页面。
