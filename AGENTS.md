# 邻里互助项目 AGENTS

本文件是本仓库的项目级强约束。所有进入本仓库的 Codex 会话、开发者和自动化任务，必须优先阅读并遵守本文件。

## 1. 项目定位

- 项目中文名：`邻里互助`
- 项目英文名：`LinBang`
- 项目类型：本地生活 / 邻里互助 / 交易撮合平台

这是一个强交易、强审核、强资金、强风控的平台系统，不是简单信息展示系统。

## 2. 仓库与框架

Codex 注意：

- 后端必须基于 `https://github.com/YunaiV/ruoyi-vue-pro`
- 管理端必须基于 `https://github.com/yudaocode/yudao-ui-admin-vue3`

绝对禁止：

- 禁止拉错成 `yudao-boot-mini`
- 禁止拉错成 `yudao-cloud-mini`
- 禁止拉第三方 `ruoyi-vue-pro` 分叉仓库
- 禁止自行起另一套后台框架

说明：

- `ruoyi-vue-pro` 是芋道官方主线单体后端项目
- `yudao-ui-admin-vue3` 是芋道官方 Vue3 管理端项目
- 这两者是本项目唯一允许的标准组合

## 4. 开发顺序强约束

禁止先堆壳页面再补规则。

必须按以下顺序推进：

1. 业务规则
2. 数据模型
3. 接口契约
4. 后端实现
5. 管理端 / 用户端页面接入真实接口
6. 联调
7. 测试

## 5. 页面交付标准

页面不能只做成“架子”。

一个页面只有同时满足以下条件，才可视为完成：

1. 接入真实接口
2. 使用真实字段渲染
3. 覆盖真实业务状态
4. 覆盖真实权限控制
5. 支持真实错误提示
6. 不依赖长期 mock 数据

以下情况不算完成：

- 页面能打开但是假数据
- 表格能展示但按钮无真实动作
- 弹窗能打开但不改变业务状态
- 详情页字段与后端真实返回不一致

## 6. 接口与分层规则

后端必须严格按芋道标准分层：

- Entity
- BO
- VO
- Mapper
- Service
- Controller

接口路径必须分离：

- 用户端：`/app-api/*`
- 管理端：`/admin-api/*`

接口风格必须参考芋道 `System` 模块：

- 分页接口
- 详情接口
- 新增接口
- 修改接口
- 删除接口
- 导出接口
- 审核接口

禁止：

- 自造另一套响应结构
- Controller 直接拼业务流程
- 用 Map 充当主业务对象

## 6.1 接口文档落地规则

开发过程中，接口文档的协作基准统一使用 `Apifox`，不再以 `Swagger` 页面作为交付标准；但强制落地点首先是后端代码中的 `OpenAPI` 能力，必须保证接口信息完整、可稳定导入。

强制要求：

- 所有新增接口、改动接口、联调接口，必须先在后端代码中补齐可导入 `Apifox` 的 `OpenAPI` 信息；如任务要求落库到 `Apifox`，再同步导入
- 前后端联调、测试验收、字段确认、状态值确认，统一以 `Apifox` 中的接口定义为准
- `Apifox` 中的路径、方法、请求参数、响应字段、错误码、枚举值、示例，必须与真实后端实现保持一致
- 如果后端代码、注解、`OpenAPI` 产物与 `Apifox` 不一致，必须优先修正文档或实现，禁止带着不一致继续开发
- 后端必须提供可稳定导入 `Apifox` 的 `OpenAPI` 数据源；`Apifox` 不得长期依赖手工逐字段维护

接口字段完整性强制要求：

- 每个接口的 `OpenAPI` 产物必须维护清楚：接口用途、请求字段含义、响应字段含义、嵌套对象字段含义、分页字段含义；如导入 `Apifox`，导入结果也必须保持这些信息完整
- 所有状态字段、审核字段、启停字段、类型字段、来源字段、方式字段、结果字段，必须写明可选值语义；禁止只写“状态”“类型”“结果”这类空描述
- 所有枚举或字典字段，必须维护到至少“字段业务含义 + 常见取值 + 取值说明”的粒度；若为动态字典，也必须明确“按什么业务字典展示”
- 所有金额、数量、比例、分值、经纬度、时间、布尔字段，必须写明单位、精度、业务口径或真假含义
- 所有关联 ID、业务 ID、外键字段，必须写明其关联对象，禁止只给字段名不说明指向关系
- 所有列表项、详情对象、嵌套对象、日志对象、统计对象，必须补齐字段说明，不能只维护顶层对象
- 对前端渲染有影响的展示字段、状态文案字段、状态名称字段、错误原因字段、驳回原因字段、审核备注字段，必须维护清楚用途
- 对外宣称“接口已完成”前，不允许存在前端无法根据 `Apifox` 判断字段用途、状态流转、枚举含义的情况

后端代码落地要求：

- 后端 `Controller`、`BO`、`VO` 必须通过 `@Operation`、`@Schema` 等 OpenAPI 注解承载接口文档语义，确保生成的 `OpenAPI` 可直接导入 `Apifox`
- 对重复出现的状态/枚举说明，必须优先抽取共享常量统一维护，禁止多个 VO 各自手写并逐渐漂移
- 新增接口或修改接口时，默认同时补齐请求、响应、状态值、字段说明、示例；禁止只补路径和字段名
- 如果一个接口在 `OpenAPI` / `Apifox` 中看起来还需要前端逐个猜字段、逐个问状态、逐个确认枚举，则视为该接口文档未完成

明确约束：

- `Swagger UI`、`Knife4j`、`springdoc OpenAPI` 仅可作为后端开发期辅助观察或导出能力，不再作为接口文档的最终落地物
- 禁止以“Swagger 上能看到”替代 `Apifox` 维护
- 禁止只改代码却不维护可导入的 `OpenAPI` 信息，也禁止只改 `Apifox` 不落真实接口
- 禁止前端、测试、产品分别维护多套互相冲突的接口定义

交付口径：

- 需求评审后先补代码侧接口契约，再生成可导入 `Apifox` 的 `OpenAPI` 产物；如项目流程要求，可再导入 `Apifox`
- 接口变更时先更新代码中的 `OpenAPI` 语义，再改后端与前端；如当前任务包含 `Apifox` 同步，再执行导入
- 对外说明接口是否“已完成”时，必须至少满足：后端已实现、代码中的 `OpenAPI` 已完整、可稳定导出/导入；若本次任务明确包含 `Apifox` 同步，则还需完成 `Apifox` 更新
- 对外说明接口是否“可交给前端开发”时，必须同时满足：前端无需额外逐字段追问状态值、字段含义、枚举语义、嵌套对象用途，即可按 `Apifox` 完成开发与联调

## 6.2 Apifox 桌面操作规则

当仓库任务明确涉及 `Apifox` 刷新、核对、导入、绑定数据源，且用户本机桌面应用已经打开时，允许直接使用桌面自动化能力操作 `Apifox`，不要求用户手工逐项点击。若任务仅要求维护代码侧 `OpenAPI` 能力，则不强制执行 `Apifox` 人工刷新验证。

强制要求：

- 优先复用已打开的 `Apifox` 桌面应用和当前已绑定的数据源，不重复手工新建项目、重复手工录入接口
- 进入 `Apifox` 后，必须先核对当前窗口标题、当前项目名称、当前数据源名称，确认操作对象正确后再执行刷新或导入
- 如存在多个 `Apifox` 项目标签页，必须先切换到目标项目，再执行 `项目设置 -> 绑定数据源 -> 立即导入`
- 如本次执行了 `Apifox` 刷新，则必须核对导入结果，至少确认目标数据源名称和最近导入时间已更新
- 执行桌面自动化前后，必须向用户明确说明当前在做什么，例如“正在切换项目”“正在刷新数据源”“正在核对导入时间”
- 如用户通过 `Esc` 或其他方式中断桌面自动化，必须立即停止，并明确告知本轮未完成的步骤

禁止：

- 禁止在未核对项目名称的情况下，直接对当前前台 `Apifox` 窗口执行导入
- 禁止把桌面手工点点点作为主维护方式；桌面操作仅用于刷新、核对、导入，不替代后端 `OpenAPI` 注解维护
- 禁止在 `Apifox` 中长期手工逐字段补请求、响应、状态、枚举，除非后端 `OpenAPI` 暂时无法承载且用户明确同意
- 禁止刷新错项目、错数据源后仍宣称同步完成

标准流程：

1. 确认用户已打开目标 `Apifox` 项目，或先在已打开窗口中定位目标项目
2. 核对当前项目是否为本次要同步的项目
3. 进入 `项目设置 -> 绑定数据源`
4. 对目标数据源执行 `立即导入`
5. 核对最近导入时间、导入成功状态、目标数据源名称
6. 将刷新结果回告用户

交付标准：

- 如果任务说明包含“刷新 Apifox”“同步接口到 Apifox”“帮我在 Apifox 更新”，则默认应直接执行桌面自动化刷新，而不是让用户自己点
- 如果任务未明确要求人工刷新 `Apifox`，则以代码中的 `OpenAPI` 能力完整、可导入、信息完整为强制标准，不把人工进入 `Apifox` 逐次验证作为默认硬门槛
- 只有当后端 `OpenAPI` 已更新，且本次任务明确要求 `Apifox` 同步时，才需要在 `Apifox` 刷新并核对导入时间后对外说明“Apifox 已同步完成”

## 7. 支付与地图强约束

支付能力：

- 微信支付
- 支付宝支付
- 银行卡提现/绑卡

支付实现要求：

- 必须复用芋道框架自带 `yudao-module-pay`
- 禁止重复自研通用支付中心
- 钱包、分账、提现作为业务模块扩展

地图能力：

- 统一使用高德地图

落地要求：

- 前端：高德 `JS API`
- 后端：高德 `Web 服务 API`

## 8. 核心业务规则不可绕过

必须遵守 `spec/business-rules.yaml` 中的以下规则：

- 用户与服务商角色边界
- 订单状态机
- 拆分单元状态机
- 支付、退款、提现状态机
- 区域合作商数据权限
- 风控配置项
- 分账规则配置项

特别说明：

- 拆分引擎是本项目核心功能，不能简化成单字段标记
- 主订单与单元订单必须联动
- 锁单、解锁、退款、申诉必须留日志

## 9. 联调规则

联调前必须先确认：

1. 接口路径已定
2. 请求字段已定
3. 返回字段已定
4. 状态值已定
5. 字典值已定
6. `Apifox` 文档已同步
7. 字段说明、状态说明、枚举语义、嵌套对象说明已补齐，前端无需二次猜测

禁止：

- 前端按猜测接接口
- 管理端和用户端各用一套状态文案
- 后端先返回临时字段名，后面再改
- 只看 `Swagger` / `Knife4j` 页面而不核对 `Apifox`

## 10. 会话工作规则

新会话进入本仓库后，默认第一步必须：

1. 读取本 `AGENTS.md`
2. 读取 `spec/business-rules.yaml`
3. 读取 `spec/openapi-structure.md`
4. 读取 `spec/api-contract-core.yaml`
5. 明确本仓库接口文档以 `Apifox` 为落地基准，不以 `Swagger` 为交付基准

如需新增模块或页面，必须先检查：

- 是否已有对应业务对象
- 是否已有对应状态机
- 是否已有对应接口前缀与命名规范

如果没有，先补规则，再写代码。

如涉及接口新增、接口变更、联调排查，还必须先检查：

- `Apifox` 中是否已有对应接口定义
- `Apifox` 中的字段、状态、错误码是否已同步到当前最新规则
- 当前开发任务是否会导致 `Apifox` 与真实实现偏离；若会，先补齐文档再继续

## 11. 当前阶段最优先产物

当前最优先推进顺序：

1. 全局业务规则一致性校验
2. 数据库表结构与业务 SQL
3. 接口清单与字段契约
4. 后端核心模块实现
5. P0 页面接入真实接口

如果规则、表结构、接口清单未完成，不建议大面积扩展页面开发。
如果后端真实接口未完成，也不建议继续堆长期 mock 页面。
